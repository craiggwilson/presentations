Questions to Ask Yourself

  • Will your application be read or write intensive?
  • How do you anticipate your documents changing over time?
  • Will your documents grow over time?
  • Do you anticipate needing to shard your data?

Schema - Book


book = {
  _id: "int",
  title: "string",
  *author: "ObjectId",
  isbn: "string",
  publisher: {
    name: "string",
    date: "timestamp",
    city: "string"
  },
  available: "boolean",
  pages: "int",
  summary: "string",
  subjects: ["string"]
  notes: [{
  	*user: "int",
  	note: "string"
  }],
  language: "string"
}

Schema - Author


author = {
	_id: "int",
	first_name: "string",
	last_name: "string"
}

Schema - User


user = {
	_id: "int",
	username: "string",
	signup_date: "UTC DateTime"

Applied Principles and Patterns

Principle: Schema-less does NOT mean No-Schema

  • Application-enforced Schema
  • Just because you can doesn't mean you should.

Principle: Use Rich Documents

  • Pattern: Embedded Documents 
    
book.publisher : 
{
	publisher_name : "Everyman’s Library",
	date : ISODate("1991-09-19T00:00:00Z"),
	city : "London",
},
  • Pattern: Embedded Arrays 
    
book.subjects : ["Love stories", "1920s", "Jazz Age"],
  • Pattern: Embedded Arrays of Documents 
    
book.notes : [
	{ user : 2, note : "One of the best..."},
	{ user : 3, note : "It’s hard to..."}
],

Pattern: Linking

  • Top-Level 
    
book.author : 1,
  • Embedded 
    
book.notes[0].user : 2
